# 阶梯制阈值策略详解

## 为什么需要阶梯制

前端安全审计面对的网站规模差异巨大：
- **小型网站** — 10 个 JS 文件，可完整处理
- **中型网站** — 300 个 JS 文件，需要策略选择
- **大型网站** — 1000+ 个 JS 文件（如 example.com），必须大幅裁剪

阶梯制根据 JS 数量自动调整策略，平衡**完整性**和**效率**。

---

## Phase 2: 递归提取阶梯制

| URL 数量 | 递归轮数 | 说明 |
|----------|----------|------|
| ≤200 | 3 轮 | 从已下载 JS 中提取更多 JS URL，深度探索 |
| 200-500 | 2 轮 | 适度探索，控制 URL 数量 |
| 500-800 | 1 轮 | 仅基础递归，防止爆炸 |
| >800 | 0 轮 | 跳过递归，直接处理现有 URL |

**递归提取原理：** 从已下载的 JS 文件中搜索 `src="..."` 或 `import("...")` 引用，发现更多 JS 资源。Cloudflare 防护网站常常有动态加载的 JS 不在首页 HTML 中。

---

## Phase 5: JS 美化阶梯制

js-beautify 处理大文件非常慢，且内存占用高。

| JS 数量 | 策略 | 大小阈值 | 说明 |
|---------|------|----------|------|
| ≤100 | 全部美化 | 无 | 完整性优先 |
| 100-300 | 过滤小文件 | >10KB | 跳过库文件碎片（jquery.min.js 等） |
| 300-500 | 只处理核心文件 | >50KB | 只处理 bundle 文件，跳过所有库 |
| >500 | 跳过美化 | N/A | 直接复制原文件到 js_beautified/ |

**为什么按大小过滤？**
- 小文件（<10KB）通常是库文件的碎片或 polyfill，不含业务代码
- 大文件（>50KB）通常是业务 bundle，包含配置和逻辑

---

## Phase 6: 凭据扫描阶梯制

### v2 全量模式（≤100 JS）

执行所有分析方法：
1. 对象赋值模式（OBJ_ASSIGN）
2. 环境变量引用（ENV_REF）
3. URL 嵌入凭据（URL_CRED）
4. JSON 配置块（JSON_CONFIG）
5. Base64 候选（BASE64）
6. 高置信度正则（HIGH_CONF）
7. 深度正则（keyword.json）

**输出：** 6 个 JSON 文件 + 1 个报告

### v2 过滤大文件（100-300 JS）

与全量模式相同，但跳过 >1MB 的文件：
- 超大文件通常是完整的库文件（如 react-dom.production.min.js）
- 这些文件几乎不包含业务配置
- 跳过它们避免内存溢出和超长处理时间

### v2 轻量模式（300-500 JS）

只执行快速分析方法：
1. 对象赋值模式（OBJ_ASSIGN）— 快速正则匹配
2. 环境变量引用（ENV_REF）— 快速正则匹配
3. 高置信度正则（HIGH_CONF）— 精确匹配，速度快

**跳过：**
- JSON 配置块解析（需要 AST 解析，慢）
- Base64 熵计算（需要大量计算）
- URL 嵌入凭据（需要 URL 解析）
- 深度正则（行数多，慢）

### grep 轻量模式（>500 JS）

仅对前 200 个文件执行关键词匹配：
- 关键词：api_key, secret, password, token, credential, auth
- 每个文件只扫描前 100 行（避免处理超大文件）
- 快速 grep，不生成结构化 JSON

**为什么只扫前 200 个文件？**
- 500+ 个 JS 文件通常是大型网站，大部分文件是库文件
- 前 200 个文件（按大小排序）通常包含业务代码
- 后面的文件（如 react-dom、lodash、moment）几乎不含业务凭据

---

## 内存限制策略

| 场景 | 限制 | 说明 |
|------|------|------|
| 单文件大小 | >1MB | 跳过美化 |
| 单文件大小 | >5MB | 跳过 Python 扫描 |
| 文件总数 | >500 | 跳过 js-beautify |
| URL 总数 | >800 | 跳过递归提取 |
| 递归深度 | 3 轮 | 最大递归深度（防止无限循环） |

---

## 自定义阈值

如需调整阈值，编辑 `js-harvester.sh` 中的变量：

```bash
# 递归阈值
RECURSIVE_THRESHOLD_HIGH=800    # >800 跳过递归
RECURSIVE_THRESHOLD_MID=500     # 500-800 执行 1 轮
RECURSIVE_THRESHOLD_LOW=200     # 200-500 执行 2 轮

# 美化阈值
BEAUTIFY_THRESHOLD_HIGH=500     # >500 跳过美化
BEAUTIFY_THRESHOLD_MID=300      # 300-500 只美化 >50KB
BEAUTIFY_THRESHOLD_LOW=100      # 100-300 只美化 >10KB

# 扫描阈值
SCAN_THRESHOLD_HIGH=500         # >500 使用 grep 轻量模式
SCAN_THRESHOLD_MID=300          # 300-500 使用 v2 轻量模式
SCAN_THRESHOLD_LOW=100          # 100-300 使用 v2 过滤大文件模式
```

## 不同规模网站的处理策略

### 小型网站（≤50 JS）
- 全量递归（3 轮）
- 全部美化
- v2 全量扫描
- 框架解析全量
- **预计时间：** 5-10 分钟

### 中型网站（50-300 JS）
- 2 轮递归
- 过滤小文件美化
- v2 过滤大文件扫描
- 框架解析全量
- **预计时间：** 15-30 分钟

### 大型网站（300-1000 JS）
- 1 轮递归（或 0 轮）
- 只美化核心文件（>50KB）
- v2 轻量模式
- 框架解析全量
- **预计时间：** 30-60 分钟

### 超大型网站（>1000 JS）
- 0 轮递归
- 跳过美化
- grep 轻量模式（前 200 个文件）
- 跳过框架解析（或只解析最大的几个文件）
- **预计时间：** 10-20 分钟（轻量扫描）

---

## 效率建议

1. **先轻量扫描**：对于未知规模的目标，先快速扫描（指定轻量模式）了解规模，再决定是否全量扫描
2. **指定技术栈**：如果已知技术栈，手动指定（如 `vue`），避免框架探测阶段
3. **限制域名**：如果域名发现过多子域名，手动编辑 `domains.txt` 只保留感兴趣的域名
4. **使用 `--cookie`**：对于 Cloudflare 重度防护目标，手动提供 Cookie 后可获得更完整的 JS 列表
